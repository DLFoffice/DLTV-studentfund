# รายงานตรวจระบบ DLTV Fund — v13 → v14

ตรวจไฟล์ทั้งหมด 30 ไฟล์ (HTML 5 · JS 20 · CSS 1 · Rules 1) พบประเด็น 14 ข้อ
แบ่งเป็น **วิกฤต 3 · สูง 4 · กลาง 5 · ต่ำ 2** และแก้ไขให้แล้วทั้งหมด

---

## 🔴 วิกฤต — ข้อมูลเด็กรั่วได้จริง

### C1. ใครก็ตามที่สมัคร Firebase Auth ได้ = "ครู" ทันที
**ไฟล์:** `firebase/firestore.rules` (v3/v5)

```
allow read: if signedIn() && (isAdmin() || !isStudentAcct() || ...)
```

`isStudentAcct()` เช็คว่า *มีเอกสาร `accounts/{uid}` และ role == 'student'*
ดังนั้น **บัญชีที่ยังไม่มีเอกสาร `accounts` จะได้ `!isStudentAcct() == true` = ครู**

โจมตีได้จริงด้วย 3 บรรทัดใน console ของหน้า `register.html` (apiKey อยู่ในโค้ดฝั่ง client อยู่แล้ว):

1. `createUserWithEmailAndPassword(...)` → มีสิทธิ์ signedIn
2. ไม่ต้องเขียนเอกสาร `accounts` เลย
3. `getDocs(collection(db,'students'))` → ได้ **เลขบัตรประชาชน + เลขบัญชีธนาคาร + ที่อยู่ ครบ 72 คน** และเขียนทับได้ด้วย

`register.html` เดิมก็เดินตามลำดับนี้พอดี (สร้าง Auth → แล้วค่อยอ่าน `students` → แล้วค่อยเขียน `accounts`)
ช่วงกลางนั้นคือช่วงที่บัญชีมีสิทธิ์ครูเต็มรูปแบบ

**แก้:** Rules v6 นิยาม `isStaff()` แบบ *fail-closed*
```
isStaff() = อยู่ใน admins/{email}  หรือ  accounts.role == 'staff' (แอดมินตั้งให้เท่านั้น)
```
บัญชีที่ไม่มีเอกสาร `accounts` = ไม่มีสิทธิ์อะไรเลย

### C2. นักเรียนเลื่อนขั้นตัวเองเป็นครูได้
**ไฟล์:** `firestore.rules`

```
match /accounts/{uid} { allow create, update: if isAdmin() || request.auth.uid == uid; }
```

นักเรียนแก้เอกสารบัญชีตัวเองได้ทุกฟิลด์ → ตั้ง `role = 'teacher'`
→ `isStudentAcct()` เป็น false → กลายเป็นครู เห็นข้อมูลทุกคน
หรือแก้ `studentId` ไปชี้เอกสารเพื่อน แล้วเขียนทับได้

**แก้:** v6 ล็อกฟิลด์ `role / no / username / email / name` ห้ามเจ้าของแก้
และ `studentId` เปลี่ยนได้เฉพาะเอกสารที่ **"ลำดับ" ตรงกับบัญชี** เท่านั้น
(`get(/students/$(sid)).data.no == resource.data.no` — `get()` ใน Rules ไม่ติด read rules)

---

### C3. บัญชีที่ไม่มี `no` หรือ `name` เห็นข้อมูลคนอื่นได้
**ไฟล์:** `js/18-firebase-bridge.js`

```js
if (!bindingValid && d.exists) bindingValid = true;   // ← ยกเลิกการตรวจชื่อทั้งหมด
```
คอมเมนต์บอกว่า "เอกสารผ่านการตรวจชื่อแล้วเท่านั้นถึงมาถึงจุดนี้" แต่ไม่จริง —
ถ้า `accData.no == null` (บัญชีเก่า) บล็อกตรวจชื่อทั้งก้อนถูกข้าม แล้วบรรทัดนี้ปล่อยผ่าน

**แก้:** เปลี่ยนเป็น `if (!bindingValid && d.exists && boundNameOk) bindingValid = true;`
และ `studentMode` ไม่ต้องรอ `studentId` อีกต่อไป (เดิมบัญชีที่ผูกไม่สำเร็จหลุดไปเป็นโหมดครู)

## 🟠 สูง

### H1. บันทึก 1 ครั้ง = เขียน Firestore 72 ครั้ง + ลบฟิลด์ทิ้ง
`pushAllToCloud()` เขียน **ทุกคน** ด้วย `{merge:false}` ทุกครั้งที่ `saveToStorage()` ถูกเรียก
กรอกแบบฟอร์มไม่กี่หน้า (autosave ทุก 600ms) ก็ชน quota ฟรีเทียร์ 20,000 writes/วัน
และ `merge:false` = ฟิลด์ที่เวอร์ชันนี้ไม่รู้จักถูกลบ (คือกลไกเดียวกับที่ทำให้ **ลำดับ 1 หายไป** ตามที่บันทึกไว้ใน README v13)

**แก้:** เก็บลายเซ็น `stableStringify` ของแต่ละเอกสาร → เขียนเฉพาะเอกสารที่เนื้อหาเปลี่ยนจริง และเปลี่ยนเป็น `{merge:true}`

### H2. `d.data()` ทิ้ง doc id → รหัสเอกสารกับฟิลด์เลขบัตรไม่ตรงกัน
ต้นตอของบั๊กชุด v10–v13 ทั้งหมด แอปอ่านเอกสารแล้วใช้ `s.id` (ฟิลด์) เป็นรหัสเอกสารตอนเขียนกลับ
พอครูแก้เลขบัตร → เขียนไปสร้างเอกสารใหม่ ทิ้งของเก่าค้าง

**แก้:** แนบ `_docId: d.id` ทุกจุดที่อ่านจากคลาวด์ และใช้ `_docId` เป็นรหัสเอกสารเสมอ (`cleanForCloud` ตัดออกก่อนเขียน)

### H3. `getLatestGpa()` = "ตัวสุดท้ายในอาร์เรย์" ไม่ใช่ "ภาคเรียนล่าสุด"
ครูเพิ่ม GPA ย้อนหลัง (เช่น กรอก 1/2568 หลัง 2/2568) → แดชบอร์ด / กราฟ / ระดับความเสี่ยง / แผนที่ ใช้ค่าผิดทั้งหมด
ซ้ำร้าย `saveSemData` ในโหมดเพิ่ม GPA เขียน `s.gpa = {...rec}` ทับด้วยภาคเรียนที่เพิ่งกรอกเสมอ

**แก้:** เรียง `semGpa`/`semPayments` ตามคีย์ `ปี*10+ภาค` แล้วค่อยหยิบตัวท้าย (`js/20-patches.js`)

### H4. หน้าจอพังถ้านักเรียนไม่มีประวัติการเบิกจ่าย
`normalizeStudent()` สร้าง `s.payment` ให้ **เฉพาะเมื่อ `semPayments` ไม่ว่าง**
แต่ `renderDashboard()` และ `renderPayment()` อ่าน `s.payment.p1` ตรงๆ → `TypeError` → ทั้งหน้าไม่แสดง

**แก้:** `ensureStudentShape()` เติมรูปร่างให้ครบก่อนทุกฟังก์ชันที่วาดหน้าจอ

# 📅 ระบบใหม่: การกรอก/อัปเดตข้อมูล และรายงาน **รายภาคเรียน**

## ปัญหาเดิม
แบบฟอร์ม 1/2 ถูกเก็บที่ `student.form1` และ `student.form2` — **ชุดเดียวต่อคน**
พอขึ้นภาคเรียนใหม่ นักเรียนกรอกทับของเดิม ข้อมูลภาคเรียนก่อนหายถาวร
และ "สถานะกรอกครบแล้ว" ตัดสินจากธง `__touched` เพียงตัวเดียว (พิมพ์ตัวอักษรเดียวก็ขึ้นว่าครบ)

ส่วน `semGpa` / `semPayments` แยกภาคเรียนอยู่แล้ว แต่ไม่มี "ภาคเรียนปัจจุบัน" ที่ระบบยึดร่วมกัน

## โครงสร้างใหม่ (`js/19-term-system.js`)

```js
student.forms = {
  "1/2568": { form1:{…}, form2:{…}, m1:{submittedAt,submittedBy,progress}, m2:{…} },
  "2/2568": { … },
  "1/2569": { … }
}
```

- `student.form1` / `student.form2` ยังใช้ได้เหมือนเดิม แต่กลายเป็น **นามแฝง (alias)**
  ที่ชี้ไปยังภาคเรียนที่เปิดใช้งาน → โค้ดพิมพ์ฟอร์ม / preview / ส่ง Google Sheet **ไม่ต้องแก้แม้แต่บรรทัดเดียว**
- ข้อมูลเก่าถูกย้ายเข้าภาคเรียนแรกที่รู้จักอัตโนมัติ (ครั้งเดียว, ไม่ลบของเดิม)
- ภาคเรียนปัจจุบันซิงก์ผ่าน `settings/activeTerm` (ครูตั้ง · นักเรียนอ่าน)

## สิ่งที่ผู้ใช้เห็น

**ฝั่งครู/แอดมิน**
- แถบเลือกภาคเรียนบนหน้าแบบฟอร์ม + ปุ่ม **➕ เปิดภาคเรียนใหม่**
  (กดครั้งเดียว แบบฟอร์มของทุกคนเริ่มใหม่ ข้อมูลเก่ายังอยู่ครบ)
- เมนูใหม่ **📅 รายงานรายภาคเรียน** — เลือกภาคเรียนแล้วเห็น
  จำนวนนักเรียน · GPA เฉลี่ยของภาคเรียนนั้น · จำนวนเสี่ยงสูง · ยอดเบิกจ่าย ·
  จำนวนที่ส่งแบบฟอร์ม 1/2 · จำนวนที่ยังไม่เริ่ม
  พร้อมตารางรายคน (GPA, Δ เทียบ ป.6, ความเสี่ยง, อุปสรรค, เงินทุน, % การกรอกแต่ละฟอร์ม) และ **ส่งออก CSV**

**ฝั่งนักเรียน**
- เลือกดูภาคเรียนย้อนหลังของตัวเองได้ (แต่เปลี่ยนภาคเรียนของทั้งระบบไม่ได้)
- ปุ่ม **📤 ส่งแบบฟอร์มภาคเรียนนี้** → บันทึกเวลาส่ง + ล็อกสถานะ (กด "แก้ไขอีกครั้ง" เพื่อปลด)

**สถานะการกรอก** เปลี่ยนจาก `__touched` (จริง/เท็จ) เป็น 4 ระดับที่คำนวณจากสัดส่วนฟิลด์ที่กรอกจริง:
`ยังไม่กรอก → กรอกบางส่วน (%) → กรอกครบ → ส่งแล้ว`

# ⚠️ ขั้นตอนติดตั้ง (ทำตามลำดับ ห้ามสลับ)

1. **Firebase Console → Firestore → Rules** → วางเนื้อหา `firebase/firestore.rules` → **Publish**
2. **สร้างบัญชีครูใหม่ทั้งหมด** — v6 ไม่ถือว่า "บัญชีที่ไม่มีเอกสาร accounts" เป็นครูอีกแล้ว
   - แอดมิน: เพิ่ม document id = อีเมล ใน collection `admins` (ผ่าน Console)
   - ครูคนอื่น: หน้า `accounts.html` → **👩‍🏫 สร้างบัญชีครู (staff)**
   - ครูที่มีบัญชีอยู่แล้ว: สร้างเอกสาร `accounts/<uid>` ที่มี `role: "staff"` ใน Console
   > ⚠️ ถ้าข้ามข้อนี้ ครูจะเข้าระบบแล้วเห็นหน้าว่าง (Rules ปฏิเสธการอ่าน) — ถูกต้องตามที่ออกแบบ
3. `accounts.html` → **📢 เผยแพร่ทะเบียนสมัคร (roster)** (ไม่ทำ นักเรียนสมัครเองไม่ได้)
4. เปิดแอป → เมนู **📅 รายงานรายภาคเรียน** → กด **➕ เปิดภาคเรียนใหม่** เมื่อขึ้นเทอมใหม่

## ตรวจสอบว่าปิดช่องโหว่แล้ว
เปิด console ที่หน้า `login.html` (ยังไม่ล็อกอิน) แล้วรัน:
```js
firebase.auth().createUserWithEmailAndPassword('test'+Date.now()+'@x.com','pass123')
  .then(()=>firebase.firestore().collection('students').get())
  .then(s=>console.log('❌ ยังรั่ว! อ่านได้',s.size,'คน'))
  .catch(e=>console.log('✅ ปลอดภัย:',e.code));
```
ต้องได้ `✅ ปลอดภัย: permission-denied`

| 📋 พี่เลี้ยง & แบบฟอร์ม | พี่เลี้ยง · สถานะแบบฟอร์มทุกภาคเรียน |

**ไทม์ไลน์ภาคเรียน** คือหัวใจของแท็บภาพรวม — แต่ละภาคเรียนเป็นการ์ดเรียงตามเวลา
แสดง GPA · ระดับความเสี่ยง · เงินทุนที่เบิก · สถานะแบบฟอร์ม 1/2
ภาคเรียนที่กำลังใช้งานถูกไฮไลต์ ทำให้เห็นพัฒนาการของเด็กคนหนึ่งได้ในหน้าจอเดียว

**ข้อมูลที่ยังขาด** ตรวจ 11 รายการที่จำเป็นต่อการทำสัญญาทุนและรายงาน
แต่ละรายการที่ขาดเป็นปุ่มกดกระโดดไปยังแท็บที่ต้องกรอก พร้อมแถบความครบถ้วนเป็น %

## แนวทางเทคนิค
ไม่ย้ายตรรกะการบันทึกข้อมูลเลย — ยังใช้ `renderSchoolPanel` · `renderGpaPanel` · `renderPaymentPanel`
และ `saveAndRefresh()` เดิมทั้งหมด เพียงจัดกลุ่มการแสดงผลของ `st-panel-0…7` ใหม่
`switchTabByName()` เปลี่ยนเป็นจับคู่ด้วยคำสำคัญ (`'GPA'`, `'ธนาคาร'` …) จึงไม่พังเมื่อเปลี่ยนชื่อแท็บ

สีและตัวอักษรใช้ตัวแปรเดิมใน `:root` ทั้งหมด ไม่มีสีใหม่ ปรับเฉพาะจังหวะการอ่าน
(หัวข้อทุกกลุ่มใช้รูปแบบเดียวกันแทนที่จะสลับสีฟ้า/เขียว/ส้มไปมา)
